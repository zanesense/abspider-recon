Reconnaissance Modules

Every recon module in ABSpider Recon follows a standardized interface: isTested, moduleError, and (for active modules) a vulnerable flag. The ScanResults object aggregates all module outputs into a single typed container.

Module categorization

Passive modules gather data without directly probing the target or use third-party aggregators. Active modules perform direct, heuristic-based testing — only run them on authorized systems. Utility modules are helper tools for analysis.

Passive Reconnaissance

Passive modules gather information without directly probing the target, or by using third-party aggregators. These form the Basic Recon and Network Intelligence categories in the UI.

Site Information

Basic site metadata captured from the initial HTTP response. Always the first module to run.

  • Status code, response time, redirect chain
  • Page title, server banner, IP address
  • CMS identification and robots.txt analysis

Passive CLI: siteInfo

HTTP Headers Analysis

Evaluates security-relevant response headers and produces a 0–100 score plus a list of recommendations.

  • Scores Content-Security-Policy, Strict-Transport-Security, X-Frame-Options, Referrer-Policy, Permissions-Policy
  • Detects missing or misconfigured headers
  • Identifies technology hints (server, framework, CDN)

Passive CLI: headers

WHOIS / RDAP Lookup

Domain registration intelligence via RDAP, with WHOIS as a fallback.

  • Registrar, registrant, and admin contacts (when redacted, marks the field accordingly)
  • Nameserver list, creation and expiry dates
  • Domain status codes (clientTransferProhibited, etc.)

Passive CLI: whois

GeoIP Location

Pinpoints the physical (or logical) location of the target's resolved IP.

  • Country, region, city, postal code, latitude / longitude
  • ASN, ISP, and organization
  • Timezone and connection type

Passive CLI: geoip

DNS Records

Full DNS enumeration using Google DNS-over-HTTPS to avoid resolver-side filtering.

  • A and AAAA records
  • CNAME, MX, NS, TXT (SPF / DKIM / DMARC hints), SOA

Passive CLI: dns

MX Records

Dedicated mail-server analysis.

  • Mail-exchange preference order and reachability
  • SPF record validation
  • DMARC policy analysis (p=none / quarantine / reject)

Passive CLI: mx

Subnet

Calculates IPv4 network parameters when an IP is available from another module.

  • Network address, broadcast address, usable host range
  • Wildcard mask, total / usable host count
  • CIDR notation summary

Passive CLI: subnet

Subdomain Enumeration

Combines a curated wordlist with Certificate Transparency log mining via crt.sh to surface subdomains that may not be in your wordlist.

  • DNS wordlist brute-force with bounded concurrency
  • crt.sh certificate transparency mining
  • Optional SecurityTrails results when an API key is configured
  • Wildcard detection and active resolution verification

Passive CLI: subdomains

Reverse IP Lookup

Identifies other domains hosted on the same IP — useful for shared-hosting audits and adjacent-asset discovery.

  • Shared-hosting detection
  • Virtual host enumeration
  • Related domain discovery

Passive CLI: reverseip

VirusTotal

Domain reputation analysis backed by 70+ detection engines. Configure the dashboard key in Settings or set VIRUSTOTAL_API_KEY for the CLI.

  • Malicious / suspicious / clean detection ratio
  • Categories (phishing, malware, C2, etc.)
  • Last-analysis timestamp and historical reputation

Passive CLI: virustotal API key required

SSL / TLS Analysis

TLS handshake and certificate inspection. The CLI uses Node's built-in tls module to reach the origin even when the dashboard's browser CORS pipeline is blocked.

  • Certificate subject, issuer, validity window, and SANs
  • SHA-256 fingerprint and serial number
  • Signature algorithm and key size

Passive CLI: sslTls

Tech Stack Fingerprinting

Identifies the frontend, CMS, analytics, CDN, and server software used by the target. The dashboard combines header hints, body patterns, and known <meta> markers.

  • Framework and library detection (React, Vue, Angular, jQuery, etc.)
  • Analytics and marketing tools (GA, GTM, Mixpanel, Hotjar)
  • CDN and reverse-proxy identification

Passive CLI: techStack

SEO Analysis

Search-engine signals extracted from the rendered page.

  • Title, meta description, canonical URL, OpenGraph tags
  • Heading structure (H1–H6) and hierarchy checks
  • Internal / external link counts, image alt-text coverage

Passive CLI: seo

CDN Detection

Correlates response headers and DNS CNAME records with known CDN signatures, preserving the evidence behind every match.

Passive CLI: cdnDetection

Cloud Provider Detection

Identifies hosting and cloud infrastructure from server headers, provider-specific response headers, and DNS evidence.

Passive CLI: cloudProvider

Email Security

Evaluates SPF, common DKIM selectors, and DMARC policy, returning raw records, validation state, and configuration issues.

Passive CLI: emailSecurity

Cookie Audit

Reviews observed cookies for Secure, HttpOnly, SameSite, domain, path, and expiry controls.

Passive CLI: cookieAudit

JavaScript Inspection

Enumerates referenced scripts and extracts endpoints, source-map references, and patterns that may indicate exposed credentials or sensitive configuration.

Passive CLI: jsInspection

Email Harvesting

Finds publicly exposed email addresses in page content and metadata, normalizes duplicates, and records the discovery source.

Passive CLI: emailHarvesting

Robots and Sitemap

Parses robots.txt, follows declared sitemap locations, and records directives and URLs that can refine an authorized assessment.

Passive CLI: robotsSitemap

Active Reconnaissance

Active modules perform direct, heuristic-based testing. Every active module loads its payloads from a version-controlled JSON file (src/payloads/*.json) and respects the per-mode cap, delay, and concurrency settings.

Authorization required

Active scanning modules require explicit written authorization. Only scan systems you own or have permission to test. Unauthorized scanning may be illegal in your jurisdiction. See Security & Legal.

Port Scanning

Checks connectivity on common TCP ports with bounded concurrency. The dashboard can be extended with Shodan for richer service data.

  • Port profiles: web (80/443), common (~100), or full (~6500)
  • Service detection and basic banner grabbing
  • Open / closed / filtered / unknown classification

Active CLI: ports

SQL Injection Testing

Loads src/payloads/sqli.json and applies baseline-aware error heuristics. Detects error-based, boolean-based, and time-based blind SQLi patterns.

  • Baseline-aware comparison (avoids false positives from existing error pages)
  • Multiple payload families and quote styles
  • DB fingerprint hints (MySQL, PostgreSQL, MSSQL, Oracle, SQLite)

Active CLI: sqlinjection

XSS Detection

Loads src/payloads/xss.json and reports unencoded reflection of probe payloads in the response body or DOM sink.

  • Reflected, DOM-based, and stored XSS coverage
  • Context-aware testing (HTML body, attribute, script, URL)
  • Filter bypass techniques for common WAFs

Active CLI: xss

LFI Scanning

Loads src/payloads/lfi.json and matches known file-content markers (e.g. /etc/passwd root line) in the response body.

  • Path traversal patterns and null-byte injection
  • Common file markers for confirmation
  • Filter-evasion variants

Active CLI: lfi

WordPress

Identifies WordPress installations and surfaces common misconfigurations.

  • Version detection from meta generator and asset fingerprints
  • Common path checks: /wp-login.php, /wp-json/, /xmlrpc.php, /wp-config.php.bak
  • Plugin and theme enumeration when discoverable

Active CLI: wordpress

Broken Link Checker

Crawls the discovered links and reports unreachable destinations.

  • Internal and external link validation
  • HTTP status code and redirect chain analysis
  • Per-link response time

Active CLI: brokenLinks

CORS Misconfiguration

Sends a curated set of Origin headers and inspects the response for dangerous combinations.

  • Wildcard origin detection
  • Reflected arbitrary-origin with Access-Control-Allow-Credentials: true
  • Null-origin and subdomain trust issues

Active CLI: corsMisconfig

WAF Protection Detection

Sends a bounded sequence of HEAD requests and records rate-limit signals without performing a real flood.

  • Identifies Cloudflare, Sucuri, Akamai, AWS WAF, and generic challenge pages
  • Reports observed rate-limiting behavior (HTTP 429, Retry-After)
  • Distinguishes between detection and bypass — the module never evades controls

Active CLI: ddosFirewall

S3 Bucket Discovery

Generates a bounded set of bucket names from the target hostname and records existence, access state, listing exposure, and HTTP status.

Active CLI: s3Bucket

Git Exposure

Checks a curated set of high-risk Git paths such as /.git/HEAD and configuration files, separating confirmed exposure from blocked or absent files.

Active CLI: gitExposure

Open Redirect

Tests common redirect parameters with a controlled external destination and validates both response redirects and reflected client-side targets.

Active CLI: openRedirect

CVE Scanner

Fingerprints supported technologies and compares detected versions with the bundled CVE catalog. Matches are leads for validation, not proof of exploitability.

Active CLI: cveScanner

GraphQL

Checks common GraphQL endpoint paths, records reachability, and determines whether schema introspection is exposed.

Active CLI: graphQL

Rate Limiting

Sends 20 bounded requests and records response codes, timing, rate-limit headers, and whether the target begins blocking traffic.

Active CLI: rateLimit

CSRF Detection

Analyzes state-changing forms for token fields, same-site cookie defenses, and other anti-CSRF signals. Findings require manual workflow validation.

Active CLI: csrfDetection

Utility Modules

Helper tools that operate on data gathered by other modules.

Subnet Calculator

Standalone network-range calculation given an IP and CIDR.

  • Network, broadcast, and usable host range
  • Wildcard mask, total / usable host count

Utility CLI: subnet

WordPress Scanner (detailed)

Already described in the active section. Listed here for completeness as it is also used as a utility pass on known WordPress targets.

Scan modes

Every scan picks one of three modes that tune payload counts, request delays, and concurrency. The same mode is honored by the dashboard and the CLI.

ModeBehaviorBest for
Conservative Minimal footprint, low concurrency, longest delays. Passives always run; actives run with the smallest payload set. Initial recon, noisy production environments, and any target that has tripped WAFs in the past.
Adaptive (default) Balanced payload counts and delays. SmartScanManager adjusts intensity if it sees rate-limiting or 5xx errors. General assessments on assets you own.
Aggressive Full payload set, higher concurrency, and minimal delays. Still bounded — never infinite. Authorized penetration testing on dedicated staging environments.

Module result shape

Every active module's result includes a vulnerable flag and a baseline isTested flag. The moduleError field captures per-module failure without aborting the rest of the scan.

interface ModuleResult { isTested: boolean; // did this module attempt to run? moduleError?: string; // per-module error, not fatal vulnerable?: boolean; // active modules only // ...module-specific fields }

Extending with custom modules

The payload files under src/payloads/*.json are version-controlled. You can audit, extend, or replace them without touching the runner.

{ "id": "sqli-error-001", "category": "sqli", "vector": "query", "param": "q", "payload": "' OR '1'='1", "match": "sql syntax|mysql_fetch|ORA-|pg_query", "severity": "high" }

When adding a new module:

  1. Create a service under src/services/ that follows the existing shape.
  2. Wire it into the ScanConfig / ScanResults interfaces in src/services/scanService.ts.
  3. Add the corresponding card in the New Scan module selector.
  4. Document each new payload pattern in your PR.
