API Reference

Two surfaces expose ABSpider Recon programmatically: the same-origin /api/proxy contract (Vercel api/proxy.ts or FastAPI backend/main.py) and the CLI JSON output. The Supabase schema is documented at the bottom of this page.

Same-origin proxy

The Vercel handler in api/proxy.ts:1 and the FastAPI handler in backend/main.py forward browser requests to a target URL with permissive CORS headers. Use this path to bypass browser CORS for targets you are authorized to test.

Endpoint

GET /api/proxy?url=<encoded-target-url>

Request

Query paramRequiredDescription
url ✅ Absolute, fully qualified URL to fetch. Must parse with new URL().

Forwarded request headers

The proxy forwards a curated allowlist of headers from the original request. Anything outside the list is dropped to avoid leaking browser or auth metadata.

const allowedHeaders = [ 'accept', 'accept-encoding', 'accept-language', 'user-agent', 'authorization', 'content-type' ];

If the upstream request has no User-Agent, the proxy sets:

Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36

Response

StatusWhenBody
200 OK Upstream returned successfully (any 2xx/3xx/4xx/5xx is passed through transparently) Proxied body, status mirrored from the upstream, CORS headers added.
200 OK (preflight) Request is OPTIONS Empty body; Allow: GET, POST, OPTIONS.
400 Bad Request url query parameter is missing { "error": "Missing \"url\" query parameter" }
500 Internal Server Error Upstream fetch failed or new URL() threw { "error": "Failed to fetch target URL", "details": "..." }

Response headers

Every successful response includes:

Access-Control-Allow-Origin: * Access-Control-Allow-Methods: GET, POST, OPTIONS Access-Control-Allow-Headers: Content-Type, Authorization Access-Control-Expose-Headers: X-ABSpider-Proxy, X-ABSpider-Target-URL X-ABSpider-Proxy: vercel | fastapi X-ABSpider-Target-URL: https://example.com/robots.txt # All upstream response headers are also passed through

Example

# Fetch the robots.txt of an authorized target curl 'https://abspider-recon.vercel.app/api/proxy?url=https%3A%2F%2Fexample.com%2Frobots.txt'
# From the browser dashboard, the same call routes through the proxy when direct fetch fails fetch('/api/proxy?url=' + encodeURIComponent(targetUrl)) .then(r => r.text()) .then(console.log);
Use only for authorized targets

The proxy is not a generic web proxy. It exists to support authorized reconnaissance. Do not use it to bypass provider challenges, evade rate limits, or access content you are not entitled to read.

CORS pipeline (client)

The dashboard's corsProxy.ts module implements a three-tier fallback. Each tier is tried in order, with bounded retries.

  1. Direct fetch — attempts a standard fetch() with browser-like headers. Works for the majority of targets that return permissive CORS headers.
  2. Same-origin proxy — if direct fetch fails (CORS error or Cloudflare challenge), the request is routed through /api/proxy. The client requires the ABSpider proxy marker or rejects app-shell fallback HTML before any scan module parses it.
  3. WAF challenge helpers — small set of wafChallengeService.ts heuristics that may unblock a target. Detection only — the module never evades controls.

CLI JSON output

Pass --json to print the full result tree to stdout, or --output <file> to write a pretty-formatted copy to disk. The shape is the same whether you run a single module or every module. See the CLI reference for the full schema and a worked example.

{ "target": "example.com", "startedAt": "2026-06-07T10:00:00.000Z", "finishedAt": "2026-06-07T10:00:14.000Z", "durationMs": 14000, "mode": "adaptive", "profile": "custom", "modules": { /* per-module results */ }, "summary": { "modulesRun": 21, "modulesOk": 20, "modulesFailed": 1, "findings": 4 } }

Each entry in modules follows the ModuleResult contract: ok, data (module-specific), and optional error / durationMs.

Supabase schema (overview)

Row-Level Security is enabled on every user-scoped table. The migrations in supabase/migrations/ provision the schema, policies, and storage buckets.

TableKey columnsRLS
scans id (uuid), user_id, target, status, config (jsonb), results (jsonb), security_grade, elapsed_ms, created_at ✅
user_settings user_id, theme, scan_defaults (jsonb), proxy_list (jsonb), webhook_url ✅
user_profiles user_id, display_name, avatar_url, bio, role ✅
api_keys user_id, service_name, api_key, is_active ✅
scheduled_scans user_id, name, target, cron, config (jsonb), enabled ✅
scan_templates user_id, name, config (jsonb) ✅
notification_prefs user_id, channel, enabled, minimum_severity ✅
storage.avatars Private bucket keyed by user_id ✅

Example RLS policy

Every user-scoped table enforces the same pattern: a row is visible only when auth.uid() = user_id.

CREATE POLICY "Users can view own scans" ON scans FOR SELECT USING (auth.uid() = user_id); CREATE POLICY "Users can manage own scans" ON scans FOR ALL USING (auth.uid() = user_id);

Testing the proxy

A scripts/test-proxy.js helper is included for verifying deployments. It checks the OPTIONS preflight, a known 2xx upstream, and a 4xx upstream.

node scripts/test-proxy.js https://abspider-recon.vercel.app # Sample output ✓ OPTIONS preflight returns 200 ✓ GET example.com returns 200 ✓ GET https://httpbin.org/status/418 returns 418

Rate limits

There are no ABSpider-imposed rate limits on the proxy. The Vercel Edge function shares the platform's free-tier limits (100 GB-hours / month on the Hobby plan). For production deployments with sustained traffic, upgrade to Vercel Pro or run the function on a Cloudflare Worker.
