Deployment

ABSpider Recon is a Vite SPA, bundled documentation site at /docs/, and FastAPI backend. Vercel deploys the frontend and backend as Services; Docker and local deployments use the same backend contract. Every supported production path serves the built frontend — never the Vite dev server.

Paths at a glance

PathFront-end served fromapi/proxy supported?Best for
Vercel (recommended) Vite frontend service, including dist/docs/ ✅ — FastAPI backend service Production, zero-config, automatic previews
Docker production dist/ served by Nginx ✅ — FastAPI backend service proxied by Nginx Self-hosted behind your own infra
Docker development Vite dev server on :5000 ✅ — when started with the backend service Local containerized dev only
Self-hosted Node dist/ served by serve ❌ — host the proxy separately Plain VM / VPS without Docker
Static host (S3, Caddy, Netlify, …) dist/ uploaded as-is ❌ — bring your own runtime CDN-first deployments

Vercel (recommended)

Vercel reads vercel.json and deploys two Services:

  • Frontend: the Vite app at /.
  • Backend: backend/main.py at /api.
  • Docs rewrites: /docs, /docs/, and /docs/* resolve to the static documentation copied into dist/docs/.
  • Routing: native service prefixes send /api/* to FastAPI and other requests to Vite.

Set the Vercel project Framework Preset to Services before deploying.

# Install the Vercel CLI if you do not have it npm i -g vercel # Deploy from the project root vercel

Required environment variables

Add these in the Vercel project settings (Settings → Environment Variables):

  • VITE_SUPABASE_URL
  • VITE_SUPABASE_ANON_KEY
  • SUPABASE_SERVICE_ROLE_KEY (server-side only; required for API-key storage and provider requests)

SUPABASE_URL is an optional server-side override; FastAPI otherwise uses VITE_SUPABASE_URL. Configure provider keys and Discord webhooks per user in dashboard Settings.

No vercel dev for production

The platform builds the Vite frontend and FastAPI backend as separate Services. Use npm run dev locally for frontend development.

Docker production

The production image is a multi-stage build:

  1. deps — installs production dependencies (npm ci --omit=dev).
  2. builder — runs the strict TS build (tsc -b) and then vite build, producing dist/.
  3. runner — a lightweight nginx:alpine image that serves dist/ on container port 8080.

The Vite dev server is never started inside the image. Docker Compose starts the FastAPI backend service and Nginx proxies /api/* to it.

# Build and run the production image (profile: app) docker compose up --build app

Open http://localhost:3000. Verify you're actually serving the production build:

# Should print "nginx" in the Server header curl -sI http://localhost:3000 | grep -i server

Nginx config

nginx.conf proxies /api/* to the backend and ensures SPA routing works by falling back to index.html for unknown paths.

location / { try_files $uri $uri/ /index.html; }

Docker development

This is the only path that runs the Vite dev server, and it is opt-in via a Compose profile.

docker compose --profile dev up --build backend dev

Open http://localhost:5000. Hot module replacement and full source maps are enabled inside the container.

Do not expose the dev container publicly

The dev image runs the Vite dev server with full source maps and no rate limiting. It is intended for local development only. Always ship app for any internet-facing deployment.

Self-hosted Node

npm start runs serve (Vercel's reference static file server) against dist/, with the SPA fallback flag (-s) so deep links resolve to index.html. The Vite preview server is not used.

npm ci npm run build VITE_SUPABASE_URL=https://YOUR-PROJECT-REF.supabase.co \ VITE_SUPABASE_ANON_KEY=replace-with-your-supabase-anon-key \ PORT=5000 \ npm start

PORT defaults to 5000 if unset. The FastAPI backend is not bundled by serve; run backend/main.py separately and route /api/* to it if this deployment needs browser-side CORS fallback.

Static host (S3, Caddy, Netlify, Cloudflare Pages, …)

Any host that can serve static files works. The only caveat is the proxy: if your static host does not have edge functions, run the dashboard without the proxy and rely on direct CORS-friendly targets or the CLI.

npm run build # Upload the contents of dist/ to your static host

Porting the proxy

If your host supports serverless functions, you can port the backend/main.py proxy contract. The handler:

  1. Reads ?url= from the query string and validates the target.
  2. Forwards a curated set of headers from the original request.
  3. Sets a default Chrome User-Agent if none was provided.
  4. Returns the response body with permissive CORS headers and the upstream status code.

Pre-deployment checklist

Run before you ship
  • npm run lint — ESLint 9 with the typescript-eslint rules
  • npm run typecheck — strict tsc -b --noEmit
  • npm run build — strict TS build + Vite build
  • npm test — Vitest unit suite
  • npm run audit — npm audit --omit=dev (local pre-release only)

The GitHub Actions workflow runs lint, typecheck, build, and test on every push and pull request to main.

Custom domains

None of the deployment paths have ABSpider-specific domain rules. Point your DNS at the host:

  • Vercel — add the domain in the Vercel project, follow the DNS instructions, Vercel provisions a Let's Encrypt certificate.
  • Docker production — front the Nginx container with your reverse proxy (Caddy, Traefik, Cloudflare, or a managed load balancer).
  • Static host — follow the host's documentation. Most providers handle HTTPS automatically.

Observability

ABSpider Recon has no built-in telemetry. Recommended external observability for production:

  • Error tracking — Sentry, Bugsnag, or a self-hosted GlitchTip instance
  • Uptime monitoring — UptimeRobot, Better Stack, or Pingdom for the dashboard URL and the api/proxy endpoint
  • Logs — your platform's log drain (Vercel, CloudWatch, Stackdriver, etc.)

Rollback strategy

For Vercel, every deployment is immutable. Use the Vercel dashboard or CLI to roll back to any previous deployment.

# Roll back to the previous deployment vercel rollback # Roll back to a specific deployment vercel rollback <deployment-url>

For Docker, retag the previous image and re-deploy:

docker tag abspider-recon:previous abspider-recon:latest docker compose up -d app
