Application Architecture

ABSpider Recon is a Vite SPA backed by Supabase. The browser is the only client; the scanner modules run inside the browser for the dashboard and inside Node.js for the CLI. Both surfaces share the same module contracts.

System topology

The high-level data flow from a user-entered target to a finished report.

User (Browser or Terminal) │ ▼ Surface ├─ Dashboard (Vite + React SPA) └─ CLI (Node.js ESM, abspider-cli.mjs) │ ▼ Scan Engine ├─ ScanService (orchestration, state, lifecycle) ├─ SmartScanManager (adaptive intensity) └─ RequestManager + CORSProxy (network layer) │ ▼ Recon Modules (src/services) siteInfo · headers · whois · geoip · dns · mx · subnet · subdomains · reverseip · virustotal · sslTls · techStack · seo ports · sqlinjection · xss · lfi · wordpress · brokenLinks · corsMisconfig · ddosFirewall │ ▼ External ├─ Targets and intel sources (DNS, RDAP, CT logs, …) ├─ Supabase (auth, RLS-scoped storage, settings) ├─ /api/proxy (FastAPI backend/main.py) └─ Third-party APIs (Shodan, VirusTotal, …) — optional │ ▼ Reports PDF (jsPDF) · DOCX · Markdown · CSV · JSON

Provider tree

The application entry point wraps the entire component tree in several essential providers. The order matters: ErrorBoundary is the outermost wrapper so uncaught errors don't crash the app.

ErrorBoundary // catches uncaught React errors └─ QueryClientProvider // TanStack Query — default retry 1, no refetch on focus └─ ThemeProvider // light / dark mode toggle └─ NotificationProvider // global toasts + persistent alert center └─ TooltipProvider // accessible tooltips └─ BrowserRouter // React Router 7 context └─ <App /> // route table below

Lifecycle hooks in App.tsx

  • cleanupStuckScans() runs on mount to reconcile local state with the database, marking interrupted scans appropriately.
  • A beforeunload listener warns the user if they try to close the browser while scans are running.
  • startScheduledScanChecker() boots the client-side scheduler for recurring scans.

RequireAuth guard

The RequireAuth HOC wraps every protected route. It performs two critical functions:

  1. On mount, it calls supabase.auth.getSession() to verify an existing session.
  2. It subscribes to onAuthStateChange so a sign-out in another tab (or an expired session) immediately redirects to /login.

While checking, the component renders a Loader2 spinner to prevent content flicker. Protected pages are nested inside a single catch-all * route that mounts the persistent layout, so the sidebar and footer do not re-mount when navigating between protected views.

Landing and session behavior

The landing page at / is always available, even for authenticated users. Clicking Sign in or Open scanner checks supabase.auth.getSession(): authenticated users go to /dashboard, while unauthenticated users see the ModernLogin modal.

The login modal supports email/password, sign-up, and magic-link flows. The Remember me option writes abspider-remember-session to localStorage and a per-tab abspider-session-active marker to sessionStorage; unchecked sessions are cleared after a browser-session restart.

DashboardPage shows a one-time beta/dev-stage dialog after the user's first dashboard entry. Dismissal is stored in localStorage under abspider-beta-notice-seen, while the dialog links users to GitHub for starring the repository and reporting bugs.

ThemeContext persists the last valid light or dark choice in localStorage under abspider-theme. Invalid values fall back to dark mode until the user toggles again.

Sidebar layout & composition

Once authenticated, every page renders inside the same layout shell: AppSidebar + SidebarInset + Footer. The sidebar is wired to TanStack Query to surface live stats from scan history.

  • Navigation menu — Dashboard, New Scan, All Scans, Reports, Settings, Account.
  • Quick Stats — Active and completed scan counts, fetched via useQuery.
  • Theme toggle — Drives the global ThemeContext.
  • Profile popover — Houses the ProfileCardPopover with avatar, role, and sign-out.

Scan engine

The scan engine is the core orchestration layer. It manages the transition from a user-defined ScanConfig to an active, stateful Scan entity.

Core flow

Scans are initiated via startScan, which initializes a Scan object with a unique UUID, maps the configuration, and registers an AbortController in an in-memory activeScans Map. The engine uses a modular execution pattern where each enabled service runs in a runScan loop that updates the scan's progress and state in real time.

Key data models

InterfacePurposeKey fields
ScanConfig Defines scan parameters target, threads, useProxy, smartScanEnabled, module toggles
ScanResults Container for all module outputs siteInfo, dns, sqlinjection, sslTls, …
Scan The primary state object id, status, progress, config, results, securityGrade
ScanDbRow Database-mapped structure scan_id, user_id, elapsed_ms, smart_scan_level

Lifecycle states

A scan moves through these states. scanService.ts is the single source of truth for transitions.

start → running → completed │ │ │ ├─→ paused → running │ ├─→ stopped │ └─→ failed └─ cleanupStuckScans() on mount marks orphan running scans as failed

toScanDbRow and fromScanDbRow map between the camelCase Scan object and the snake_case scan_db rows in Supabase.

Smart scan manager

The SmartScanManager provides adaptive scanning. It analyzes a target before launching and adjusts payload counts and rate limits based on the selected mode: conservative, adaptive, or aggressive.

  • Conservative — low payload cap, high delay, single-thread. Minimal footprint.
  • Adaptive (default) — baseline-aware; reduces intensity after a target returns 5xx or 429.
  • Aggressive — full payload set, high concurrency, minimal delays. Still bounded.

The manager is integrated into the New Scan UI to simplify configuration — you pick the mode, not the numeric parameters.

Request infrastructure

All network traffic generated by the engine passes through the RequestManager. It centralizes rate limiting, retries, and metrics, and it interfaces with the CORS proxy pipeline.

Tiered request pipeline

Browser-side CORS limits are bypassed with a tiered approach:

  1. Direct fetch — attempt a standard fetch with browser-like headers.
  2. Same-origin proxy — if direct fetch fails (CORS or Cloudflare), route through /api/proxy. Vercel Services, Docker, and local backend deployments route it to FastAPI.
  3. WAF challenge helpers — small set of wafChallengeService.ts heuristics that may unblock a target without evading controls.

The same-origin proxy

The FastAPI handler at backend/main.py provides the shared deployment contract:

  1. Reads ?url= from the query string and validates it with new URL().
  2. Forwards a curated set of request headers (the allowedHeaders allowlist) to avoid issues.
  3. Sets a Chrome User-Agent if one was not provided.
  4. Returns the response body with permissive CORS headers and the upstream status code.

The proxy responds to OPTIONS preflight with Allow: GET, POST, OPTIONS, exposes X-ABSpider-Proxy and X-ABSpider-Target-URL, and lets the frontend reject misrouted SPA fallback responses before scanner modules parse them as target evidence. Use it only for hosts you are authorized to test — it is not a generic web proxy.

Scan configuration & templates

The New Scan page uses react-hook-form and zod to validate user input before a scan starts. Users can save their configurations as templates in localStorage for quick reuse.

  • Each ScanConfig includes per-mode payload counts and per-module toggles.
  • Recurring scans are stored in scheduledScans with an interval and a localStorage-backed scheduler that runs while the tab is open.
  • Validation rules live next to the form schema in src/pages/NewScan.tsx.

Security grading

The calculateSecurityGrade function processes the aggregated ScanResults to produce a numerical score (0–10) and a letter grade (A–F). The score is derived from:

  1. Critical vulnerabilities — high deductions for confirmed SQLi, XSS, or LFI.
  2. Configuration issues — deductions for missing security headers or expired SSL certificates.
  3. Information leakage — deductions for exposed server signatures or sensitive DNS records.

The final grade is displayed in the ScanStatus component using a color-coded Badge:

Authentication flow

Authentication is handled by Supabase Auth with email/password and magic-link flows.

  1. User opens the login modal from the public landing page.
  2. They sign in with email/password or request a Supabase one-time magic link.
  3. The Remember me option controls whether the session survives a browser restart.
  4. The RequireAuth guard picks up the session via onAuthStateChange and grants access to protected routes.

Sessions are JWTs managed by Supabase. The default expiry is 7 days; the SPA listens for SIGNED_OUT events and forces a redirect if a session is lost in another tab.
